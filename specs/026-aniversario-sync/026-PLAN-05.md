    1. Docstring de módulo em português: descrever propósito (migração retroativa de
       aniversários existentes para o sistema de sync da fase 026), como executar
       (docker exec makima-web python -m scripts.migrate_birthday_sync),
       comportamento idempotente (person_dates com link já existente são pulados pelo
       NOT EXISTS na query de busca), e que o hostname do PostgreSQL
       (personal-agent-makimadb-k3bxg9) só resolve dentro do container makima-web.

    2. Imports: os, logging, sys. Import lazy de komi_sync dentro de main().

    3. Função main() -> None:
       - Log de início com timestamp: "=== Migração birthday_sync — início ==="
       - Importar run_select de agents.db
       - Buscar person_dates de aniversário SEM link:
           SELECT pd.id, pd.label, pd.date::text, pd.person_id, p.name
           FROM person_dates pd
           JOIN people p ON p.id = pd.person_id AND p.deleted = FALSE
           WHERE LOWER(pd.label) LIKE '%anivers%'
             AND NOT EXISTS (
               SELECT 1 FROM birthday_sync_links bsl WHERE bsl.person_date_id = pd.id
             )
           ORDER BY pd.id
       - Logar contagem encontrada: f"Encontrados {len(rows)} person_dates sem link."
       - Para cada row: chamar komi_sync.push_person_date(row["id"])
         (push_person_date é idempotente — cria tarefa + link se não existe)
         Log por row: f"  [{i+1}/{len(rows)}] person_date {row['id']}: {row['label']} — {row['name']}"
       - Ao final: SELECT COUNT(*) FROM birthday_sync_links
         Log: f"=== Total de links após migração: {count} ==="
       - Verificar residual: SELECT COUNT(*) com a mesma query do início
         Log: f"=== person_dates de aniversário sem link restantes: {count} (esperado: 0) ==="

    4. if __name__ == "__main__": main()

    Comentários abundantes em português em cada etapa.
    O script usa DATABASE_URL via agents.db (mesma env var de todo o projeto).
